## Execution Summary

**Result: FAIL (pre-fix) → PASS (post-fix)**
**Attempts:** 2 (1 fix for sessions API format, then ran final to discover bug; then applied fix and re-ran)

**Bug discovered:** Sessions stuck in 'starting' state after server restart are NOT recovered.

**Root cause (confirmed by reading source):**
- `SessionManager.init()` (lib/sessions/SessionManager.js line 73): `const wasLive = meta.status === STATES.RUNNING;`
- Only sessions with status `'running'` are auto-resumed on startup.
- Sessions with status `'starting'` are a known, valid state (sessionStates.js line 3: `STARTING: 'starting'`).
- `'starting'` passes the known-status check at line 58-61 (not migrated to 'stopped').
- Therefore: a session with `status='starting'` loads, is stored in the Map, but `wasLive` is false → no auto-resume → status remains `'starting'` permanently.

**Impact:**
- If the server crashes while a session is transitioning STARTING→RUNNING (a ~1-5ms window), the session is stuck in `'starting'` after restart.
- The user cannot restart the session because `DirectSession.refresh()` requires status not be STARTING (throws "Cannot refresh direct session: already starting").
- The user cannot start the session (it's already "starting").
- The session is effectively bricked until manually fixed in sessions.json.

**Fix required:** In `SessionManager.init()`, add handling for `status === STATES.STARTING`:
```javascript
const wasLive = meta.status === STATES.RUNNING || meta.status === STATES.STARTING;
```
Or equivalently, before the auto-resume check, normalize `STARTING` to `STOPPED`:
```javascript
if (meta.status === STATES.STARTING) {
  meta.status = STATES.STOPPED;
}
```

**Server behavior:**
- Server does NOT crash on startup with stuck-starting session — only the session state is wrong.
- Crash log remains empty after restart.
- All other API endpoints remain functional.

**Issues encountered during implementation:**
1. Sessions API returns `{ managed, detected }` not bare array. Fixed.
