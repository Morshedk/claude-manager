Key finding — Design correction:
The design analysis predicted that concurrent refreshes from two WS connections would
be blocked by the STARTING state guard. However, the actual behavior is:

  BOTH refreshes succeeded — no session:error was returned to either WS.

This is because Node.js processes each WebSocket message as a SEPARATE async event.
The server handles ws1's refresh completely (kill PTY → spawn → STARTING → RUNNING)
before the event loop processes ws2's refresh message. By the time ws2's refresh
handler runs, the session is back in RUNNING state — which is a valid state to refresh
from. So ws2's refresh gets a second valid refresh.

The state sequence observed:
  RUNNING → starting → running → starting → running

Critical assertion — NO ORPHANED PTY:
Despite two refresh cycles completing, the ps count shows exactly 1 marker process
after stabilization. The _ptyGen mechanism correctly kills the first new PTY when
the second refresh fires. There is no orphaned process.

The test PASSES because the key invariant holds:
  "No orphaned PTY processes" — TRUE (count=1 throughout)
  "Session reaches stable running state" — TRUE
  "Server survives concurrent refresh" — TRUE
  "No unhandled exceptions" — TRUE

Deviations from design:
1. exec -a is not available on this system; used tmpDir path as ps marker instead
2. Both refreshes succeeded (design predicted one would be blocked). This is NOT a bug:
   the refreshes are sequential (Node.js single-threaded event loop), not truly concurrent.
   The design overestimated how "concurrent" WS message handling is in practice.
3. _ptyGen prevents orphaned PTYs even when two refreshes succeed sequentially — this
   is the intended safety mechanism and works correctly.
