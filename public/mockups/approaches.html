2 Snapshot-based session cards — clean output + system events, animated
Option 2A
Enhanced log file + polling
Keep the existing SessionLogManager approach but improve it. Server writes clean lines + system event markers (reconnects, start, stop, time gaps) to the log file. Client polls every 2s instead of 3s, diffs the result against the previous snapshot, and animates new lines in with staggered CSS transitions.
+ Minimal architecture change — improve existing code · − 2s polling delay before events appear · extra HTTP requests
Option 2B ★ Recommended
WebSocket-pushed snapshot per session
Server computes a snapshot object per session whenever meaningful output changes: { lines: [...], events: [...], version: N }. lines = last 30 clean terminal lines (ANSI stripped, spinners removed). events = ordered list of system events (session start, reconnect, stop, time gaps >30s). Snapshot is pushed over the existing WebSocket sessions:list broadcast — no new endpoints.

Client stores last version. When a new snapshot arrives with a higher version, it computes the diff (new lines/events since last snapshot) and reveals them with staggered animation. Session card and events panel both read from the same snapshot.

Stream illusion: Each snapshot diff is revealed line-by-line, 60ms stagger per line, slide-in from bottom. Feels live even though it's periodic.
+ Instant push (no poll lag) · single source of truth · reuses existing WS · events panel and card are consistent · − Slightly more server-side work to maintain snapshot state

3 Command send — fix bracket-paste so pressing Send actually executes
Option 3A ★ Recommended
Remove bracket-paste, send content + carriage return
One-line fix in CommandBuffer.js:31. Replace:
'\x1b[200~' + content + '\x1b[201~'
with:
content + '\r'

\r (carriage return) is what terminals use to execute a command. The block is sent whole (user chose Option A for multi-line), so a for-loop arrives intact and \r at the end executes it. Same fix in the queue drain path (line 68).
+ 2-line change · correct behaviour for all input types · no edge cases · − Removes paste protection (acceptable: CommandBuffer is explicitly a command input, not a paste target)
Option 3B
Smart mode: bracket-paste for clipboard, plain for CommandBuffer
Keep bracket-paste when the xterm terminal itself receives a clipboard paste (Ctrl+V in the terminal window), remove it only in CommandBuffer.handleSend(). More surgical — preserves the protection for raw terminal pastes while fixing the Send button.
+ Preserves bracket-paste in the terminal for clipboard safety · − More code · the terminal already handles its own paste via xterm; CommandBuffer doesn't need it
