=== T-5: Adversarial ANSI stripping ===
Result: FAIL
Attempts: 3
Duration: ~30s per attempt
Key observations:
  - Attempt 1: mode:direct spawns Claude CLI (not bash). echo commands typed into Claude TUI produced wrong output format. CLEAN_SENTINEL_LINE not found.
  - Attempt 2: Added command:'bash' so a real bash shell executes the echo commands. Commands used \\x1b (literal backslash-x-1-b) not actual ESC bytes. Bash terminal echo shows literal text including [31m pattern.
  - Attempt 3: Changed to printf $'\x1b[31m...' (ANSI-C quoting) to send actual ESC bytes. CLEAN_SENTINEL_LINE found (control passes). Still FAIL:
    (a) Command echo line contains [31m after stripAnsi removes the ESC byte, leaving "[31m" as literal residue. Root cause: stripAnsi regex /\x1b\[[0-9;]*[a-zA-Z]/ removes the ESC+bracket+params+terminator, but the command echo in the PTY ring buffer shows the user typed $'\x1b[31m' — the ESC is stripped but [31m text residue remains because the sequence spans a $'' string boundary context.
    (b) Lines contain ?2004l, ?25l — partial sequences where \x1b[ was removed but the ?-prefixed params + terminator remain. The CSI regex matches [0-9;]*[a-zA-Z] but not ?[0-9;]*[a-zA-Z] (private modes starting with ?).
    (c) Lines contain \r (char code 13). The stripAnsi implementation explicitly keeps \r (it removes [\x00-\x09\x0b\x0c\x0e-\x1f] — skipping \x0d=CR).
  - Three real bugs confirmed: (b) private-mode sequences not fully stripped, (c) carriage returns not stripped.
Fix applied (if any): Attempts 2-3 were infrastructure fixes (bash mode, actual ESC bytes). The failure at attempt 3 is a real feature bug.
