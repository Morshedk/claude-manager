# SessionLog Feature Design (v2)

**Date:** 2026-04-17
**Status:** Approved
**Scope:** v2, tmux mode (TmuxSession). Direct mode deferred.
**Ported from:** `claude-web-app/docs/superpowers/specs/2026-04-14-sessionlog-design.md`

## Summary

SessionLog provides persistent, human-readable logging of everything that appears in
tmux terminal sessions. Logs survive Claude compaction and server restarts, are tied to
the tmux session ID (`cm-<uuid8>`), and expose a split-panel viewer in the browser.

**Extended for v2:** Debug tracing events are injected inline into `.log` files at every
lifecycle transition — kill, stop, refresh, server crash, PTY attach/detach — so the log
answers "what happened and when" without needing to correlate separate crash.log files.

---

## Architecture

### New module: `lib/sessionlog/SessionLogManager.js`

Single responsibility: owns pipe-pane capture lifecycle and log processing for all tmux
sessions. Nothing else touches sessionlog files directly.

### Two-file pattern per session

```
data/sessionlog/
├── {sessionId}.raw    ← tmux pipe-pane writes here continuously (zero-loss staging)
└── {sessionId}.log    ← 20s tick strips ANSI + injects timestamps (persistent record)
```

- `.raw` — staging buffer only; truncated to zero after each successful 20s processing pass
- `.log` — append-only; never truncated; survives session end and server restarts

### Data flow

```
tmux pane output
      ↓  (pipe-pane, real-time, zero-loss)
{sessionId}.raw
      ↓  (20s tick — only if raw has grown since last pass)
strip ANSI → inject timestamp marker → append to {sessionId}.log → truncate raw
      ↓  (HTTP polling, ~3s interval)
GET /api/sessionlog/:sessionId/tail
      ↓
Browser split panel (read-only)
```

Lifecycle events (kill/stop/refresh/crash) are injected directly into `.log` at the moment
they occur — they bypass the `.raw` staging buffer entirely since they are generated by
the server, not by the tmux pane output.

## Capture Mechanism

`SessionLogManager.startCapture(sessionId, tmuxName)` runs:

```bash
tmux pipe-pane -t {tmuxName} -o 'cat >> data/sessionlog/{sessionId}.raw'
```

This pipes every byte of new pane output to the raw file in real-time — zero-loss,
no buffer overflow risk.

**On session resume/refresh:** pipe-pane stops when the PTY detaches. `startCapture`
is called again when `TmuxSession._attachPty()` completes (see lifecycle integration).

**On session stop:** `SessionLogManager.stopCapture(sessionId, tmuxName)` runs
`tmux pipe-pane -t {tmuxName}` (no `-o` flag = stops the pipe).

---

## 20-Second Tick

`SessionLogManager.tick()` is called every 20 seconds from the server (same interval
as the existing watchdog pattern).

For each active session, `tick()`:

1. Checks `.raw` file size against last-known offset
2. **If unchanged:** skip — no timestamp injected, no write
3. **If new content exists:**
   a. Read new bytes from last processed offset to EOF
   b. Strip ANSI escape codes
   c. Append to `.log`: `\n--- {ISO8601} ---\n{stripped content}`
   d. Update last-processed offset in memory
   e. Truncate `.raw` to zero bytes

The no-new-content guard prevents empty timestamp markers during idle sessions.

## Debug Lifecycle Events

These are injected directly into `.log` (bypassing `.raw`) at the moment they occur.
Format: `\n=== {EVENT} | {ISO8601} | {details} ===\n`

| Event marker | When | Details |
|---|---|---|
| `PTY:ATTACH` | `_attachPty()` succeeds | `tmuxName` |
| `PTY:DETACH` | `_wirePty().onExit` fires (PTY closed) | `tmuxName, wasRunning` |
| `SESSION:STOP` | `TmuxSession.stop()` called | `reason: explicit` |
| `SESSION:REFRESH` | `TmuxSession.refresh()` called | `tmuxAlive: true/false` |
| `SESSION:KILLED` | `onExit` fires AND `!_tmuxSessionAlive()` | `tmuxName` |
| `SESSION:RESUME` | tmux dead path in `refresh()` kicks off `start()` | `claudeCmd` |
| `SERVER:CRASH` | SIGTERM/SIGHUP received by server | `pid, signal` |
| `SERVER:START` | server started, session re-attached from store | `pid, sessionCount` |

These events require no UI — they are plain text annotations visible in the split panel
and downloadable via the full-log API. Their main use is post-mortem debugging: when a
session dies unexpectedly, the `.log` shows the exact transition that caused it.

## Browser Split Panel

A resizable split panel added to the session overlay, toggled by a button in the
session header (same pattern as the Watchdog/TODO tabs in ProjectDetail).

**Layout:** vertical split — terminal left, SessionLog right.

**Content display:** monospace, pre-formatted, scrollable. Starts at bottom (tail-anchored).
Auto-scrolls to bottom only when the user is already at the bottom.

**Polling:** `GET /api/sessionlog/:sessionId/tail?lines=500` every 3 seconds. Client
tracks last-seen line count and appends only new lines.

---

## API Endpoints

```
GET /api/sessionlog/:sessionId/tail?lines=N
  → Last N lines of {sessionId}.log (default 500)
  → 404 if no log exists yet

GET /api/sessionlog/:sessionId/full
  → Full log file streamed (for download / agent read)

GET /api/sessionlog/:sessionId/status
  → { capturing: bool, logSizeBytes: int, lastActivity: ISO timestamp }
```

## v2 Lifecycle Integration

### TmuxSession.js hooks

| Call site | Action |
|---|---|
| `_attachPty()` — after `this.pty = ptyProcess` | `sessionLog.startCapture(id, tmuxName)` + inject `PTY:ATTACH` |
| `_wirePty().onExit` | inject `PTY:DETACH`, then if `!_tmuxSessionAlive()` inject `SESSION:KILLED` |
| `stop()` — before `tmux kill-session` | inject `SESSION:STOP`, `sessionLog.stopCapture()` |
| `refresh()` — tmux-alive path | inject `SESSION:REFRESH` |
| `refresh()` — tmux-dead path (before `start()`) | inject `SESSION:RESUME` |

### SessionManager.js hooks

| Call site | Action |
|---|---|
| `init()` — after re-attaching a RUNNING tmux session | inject `SERVER:START` |

### server-with-crash-log.mjs hooks

| Signal | Action |
|---|---|
| `SIGTERM` / `SIGHUP` | inject `SERVER:CRASH` into ALL active session logs before exiting |

### server.js

Register `sessionLogManager.tick()` on a 20-second interval after `sessions.init()`.
