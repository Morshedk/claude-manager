=== T-4: Double-delete in rapid succession — idempotent delete + single log entry (AC12, Risk 4) ===
Result: PASS (after fix)
Attempts: 2
Duration: 5.4s (attempt 2)
Key observations:
  ATTEMPT 1 FAILED: Log contained 3 entries (1 created + 2 deleted) instead of 2. Real bug found.
  Root cause: Two concurrent session:delete handlers both executed `this.sessions.get(msg.id)` before
  either reached `this.sessions.delete(sessionId)` (Map.delete). JavaScript async handlers do NOT
  serialize on `await` — both handlers started while the other was awaiting sessions.delete(), so
  both read non-null meta and both wrote a 'deleted' entry to the lifecycle log.
  The review doc's analysis ("first delete removes meta from Map before log-write begins") was wrong
  about the timing — Map.delete is inside the async sessions.delete() method, so the second handler
  can read meta before the first handler's await completes.
Fix applied:
  lib/ws/handlers/sessionHandlers.js: Added `_inProgressDeletes = new Set()` to the constructor.
  In `delete()` handler: check/add session ID to set before awaiting sessions.delete(), skip meta
  capture if already in set (second handler sees it in set and captures null meta → no log entry).
  Set entry removed in finally-equivalent position after broadcast.
  ATTEMPT 2: 2 entries total (1 created + 1 deleted), server alive, no session:error. PASS.
